## Running the App

```bash
# Start server (local dev)
python app.py          # runs on http://localhost:5000, debug=False

# First-time setup (if data/training.db doesn't exist)
python -c "from app import init_db; init_db()"
python seed.py         # imports employee master data from C:\Users\yogendra.das\Desktop\Master Emp Data\

# Windows shortcut
run.bat
```

**Default credentials:**
- SPOC (per plant): username = `plantname` (e.g. `balrampur`), password = `bcml@1234`
- Central: `central` / `bcml@1234`
- Admin: `admin` / `admin@bcml`

**Environment variables:**
- `SECRET_KEY` — Flask secret (defaults to a hardcoded dev key)
- `DATABASE_PATH` — override SQLite path (used on Railway; default is `data/training.db`)

Templates auto-reload without server restart (`TEMPLATES_AUTO_RELOAD = True`). Python code changes require a restart.

## Architecture

Modular Flask app: thin `app.py` (entry point, ~50 lines) + domain route modules under `tms/routes/` + SQLite (`data/training.db`) + Jinja2 templates. No ORM — all SQL is raw with `sqlite3`. Deployed on Railway via `railway.toml`.

**Module layout:**
- `tms/constants.py` — all enums, plant list, month list, type abbreviations
- `tms/db.py` — `get_db()`, `init_db()`, startup migrations
- `tms/decorators.py` — `@login_required`, `@spoc_required`, `@central_required`, `@admin_required`
- `tms/helpers.py` — all shared helper functions (~700 lines)
- `tms/routes/` — one file per domain: `auth`, `employees`, `tni`, `programme`, `calendar`, `training`, `summary`, `central`, `export`, `api`

Each route module exposes `_register(app)` which registers routes directly on the Flask app (not Blueprints) — so all `url_for('endpoint_name')` calls in templates work unchanged.

### Role System
Three roles enforced via decorators (`@spoc_required`, `@central_required`, `@admin_required`):
- **spoc** — plant-level user; all training data entry scoped to `session['plant_id']`
- **central** — read-only cross-plant dashboard
- **admin** — can impersonate any plant

### Data Model (6 core tables)

| Table | Purpose |
|---|---|
| `employees` | Master headcount per plant; `collar` = Blue Collared / White Collared |
| `tni` | Training Need Identification — one row per emp+programme demand. `UNIQUE(plant_id, emp_code, programme_name)` |
| `programme_master` | Canonical programme list per plant; `prog_type` synced from TNI |
| `calendar` | Planned sessions (Sheet 2C planning); `session_code` = `UNIT/TYPE/NNN/YY-YY/BNN` |
| `programme_details` | Conducted session details (Sheet 2C actuals); one row per session |
| `emp_training` | Individual attendance records (Sheet 2A); one row per person per session |

### Key Conventions

**Session codes** are generated by `_new_session_code()` as `prog_code/FY/B01`, e.g. `BCM/TEC/001/26-27/B01`.

**`programme_master` is the gatekeeper.** The Programme Name field in the Calendar, 2A, and 2C forms is a searchable dropdown bound to `programme_master`. A programme that does not exist in `programme_master` cannot be selected — the form shows "No results found". This is a deliberate guard against free-text spelling variants breaking TNI matching and summary grouping. For any new programme (TNI-driven or New Requirement), it must be added to `programme_master` first before it can be used anywhere else.

**Warning — `Sync from TNI` wipes manual entries.** `programme_master_sync_from_tni()` deletes the entire master list and rebuilds it from TNI data only. Any programme added manually for a New Requirement that has no TNI rows will be deleted by a sync. Add New Requirement programmes to the master only if the SPOC understands this risk.

**Audience is locked from TNI.** `_derive_audience(plant_id, prog_name, db)` checks the `tni` table only — if both BC and WC employees are nominated → `Common`; BC only → `Blue Collared`; WC only → `White Collared`. This takes precedence over form input at every calendar write path (add, edit, bulk upload). `_sync_calendar_from_2c()` auto-corrects audience on each calendar page load.

**New Requirement audience is free.** If a programme has no TNI rows, `_derive_audience()` returns empty → the form's `target_audience` value is used as-is. The SPOC must set it manually.

**Audience mismatch distortion.** If a WC employee attends a session whose `target_audience = 'Blue Collared'` (set from TNI), their manhours land under WC in the Monthly Summary but the programme is counted as a BC programme — the two columns tell different stories. To avoid this, add the WC employee to TNI for that programme so audience auto-upgrades to `Common`.

**Source values** are exactly two: `TNI Driven` or `New Requirement`. Any other value is silently coerced to `TNI Driven` at write time (in `add_calendar` and `edit_calendar`). `_migrate_tni_source()` collapses legacy values at startup.

**New Requirement coverage impact.** Attendees of a New Requirement session are recorded in `emp_training` and count toward person seats and man-hours in the Monthly Summary. They do NOT count toward Coverage % — coverage is calculated only over employees who appear in the `tni` table for that programme.

**Programme name canonicalization** — `_canonical_prog()` fuzzy-matches raw names (cutoff 0.82) against `programme_master`. If no match is found it returns the original string unchanged (does not block saving). `_smart_title()` normalises casing using a domain vocab (e.g. "5S", "EHS", "SOP" stay uppercase).

**2C Programme Details entry flow.**
1. The Session Code dropdown is bound to `calendar` — only sessions that exist in the calendar can be recorded in 2C.
2. On save, programme name, type, level, mode, and audience are auto-copied from the `calendar` row; only dates, faculty, cost, and feedback scores are entered by the SPOC.
3. Saving a 2C record sets `calendar.status = 'Conducted'`. Deleting it resets status to `'To Be Planned'`.
4. `cal_new` is set to `'Calendar Program'` if the session exists in calendar, `'New Program'` otherwise.

**2A Training Records entry flow.**
1. Session Code is optional. If provided and found in `calendar`, programme name/type/level/mode are auto-filled; start/end dates default to calendar plan dates if not provided.
2. If no session code is given, the SPOC must supply the programme name — it is fuzzy-matched via `_canonical_prog()` against `programme_master`.
3. Employee code must exist and be active in the `employees` table for the plant.

**Monthly Summary calculation** (`_calc_summary` / `_calc_totals`):
- No. of Programmes → from `programme_details` (2C) `audience` field
- Person Seats / Man-Hours → from `emp_training` (2A) joined with `employees` for collar
- Coverage % = Cumulative trained (TNI emp who actually trained) / Fixed (all TNI nominated emp for that prog_type+collar). Non-TNI attendees count in seats but not coverage.

**TNI analysis pipeline** — `_smart_analyze_rows()` fuzzy-matches uploaded rows against master vocab, flags low-confidence matches, stores a JSON file (`data/tni_analyze_*.json`). SPOC reviews and confirms via `/tni/analyze/confirm`.

### Startup Migrations
`init_db()` (called at process start) runs:
1. `schema.sql` — `CREATE TABLE/INDEX IF NOT EXISTS`
2. `_ensure_indexes()` — adds performance indexes to existing DBs
3. `_migrate_tni_unique()` — enforces UNIQUE on tni if missing
4. `_migrate_tni_source()` — collapses old source values in tni + calendar
5. `_cleanse_master_spelling()` + `_cleanse_programme_names()` — normalises names

### TMS-Wide Consistency Rule
**When changing any field, label, enum value, or source — check and fix it across ALL templates and ALL routes, not just the touched file.** This applies to: source dropdowns, audience values, prog_type enums, collar values.

### Performance
- SQLite WAL mode enabled per connection (`PRAGMA journal_mode=WAL`)
- TNI data endpoint (`/tni/data`) uses paginated server-side fetch (30/page) with debounced search (320ms)
- `EXISTS(...)` subqueries used instead of materialised subquery joins for completed-status checks
